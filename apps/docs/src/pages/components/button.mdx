## Use Cases

### Submitting a form

The user completes a form and activates a submit button to persist their input. Use a verb–noun label naming the action and target (e.g., "Save changes", "Create account"). The button enters a busy state while the request is in flight; on success, the form closes or transitions to a confirmation view. Reserve the busy state for short async operations — for longer-running processes, use a separate notification pattern.

### Beginning a new task

The user initiates a workflow for a resource that doesn't yet exist — for example, clicking "Create device" on a device management page or "Deploy application" in a pipeline view. This is the primary action on the page; use it once per view with a verb–noun label (e.g., "Create device", "Schedule backup"). Activating it navigates to the next step or opens the first stage of the workflow.

### Providing a primary call to action

A single primary button stands out visually among all other controls, signaling the most important or recommended action on the page. For example, "Activate service" on a dashboard draws the user's eye to the recommended next step. Use at most one primary button per view; all other actions use secondary or default styling.

### Presenting secondary actions

Secondary buttons surface supporting actions that complement the primary call to action without competing with it — for example, "Download report" alongside a primary "Edit resource". Multiple secondary buttons may appear per view, each representing a distinct and meaningful action.

### Cancelling or dismissing

A cancel button gives users an escape hatch to abandon an in-progress workflow without saving — for example, closing a "Create device" form and returning to the device list. Use default styling and position per established hierarchy. Use "Cancel" as the label unless a more specific verb improves clarity (e.g., "Discard changes").

## Behaviors and States

### Interactive States

Button responds to user input through the following visual states:

- **Rest** — The button's default appearance when no interaction is occurring.
- **Hover** — A subtle background or color shift signals that the button is interactive when the pointer enters it.
- **Focus** — A visible focus ring surrounds the button when it receives keyboard focus, satisfying WCAG 2.4.7.
- **Active** — The button depresses visually while the pointer is held down or the activation key is held.

### Application States

Application logic can place a button in the following states:

- **Disabled** — Prevents user interaction and applies reduced-opacity styling. Prefer leaving buttons active and surfacing an explanation on click over hiding or disabling them; do not show a disabled button for an action that is simply unavailable in the current context — remove it or show it active with an explanatory message.
- **Busy** — Displays a spinner indicator to signal that an action is in progress. Use for short async operations (a few seconds or less); if the action may take longer, let the user leave and notify them separately when the task completes.
- **Success** — Displays a check mark after an action has completed, providing immediate in-place confirmation before the UI transitions.

### General Behaviors

#### Responsive behavior

At screen widths ≤ 576px (Grommet `small` breakpoint):

- Expand single buttons to fill the full width of their container.
- Reflow button groups vertically so each button achieves full width.
- The minimum button width for any button is 42px.
- Button label text must not scale down, wrap, or truncate at smaller widths.

#### Button alignment and ordering

Button alignment depends on the surface they appear on — modals, drawers, inline forms, and page footers each have established conventions. Consult the alignment table below for guidance by context. Within a button group, always order by importance: Primary first, followed by Secondary, then Default.

#### Buttons vs. anchors

Use a `Button` when the interaction performs an in-page action (submit, create, delete, trigger UI). Use an `Anchor` or `Link` when the result is navigation to a new URL or page. This distinction is critical for screen reader users, who rely on the element role to anticipate whether focus will move to a new document or whether something will happen within the current page.

## Content Guidelines

- Use clear and concise wording — avoid contractions or colloquialisms.
- Ensure the button label matches the action or route destination.
- Use verb + noun format for the primary action label (e.g., "Apply filters", "Create device").
- Omit the noun from labels beyond the primary button when the action is clear and unambiguous (e.g., "Reset" instead of "Reset filters").
- Use "Add" when associating an existing object with another object or set of objects (e.g., "Add device" to an existing group).
- Use "New" to open a blank template or canvas for user input before the object is committed (e.g., "New device" to start configuration).
- Use "Create" for form submission that creates a new object instance in the system (e.g., "Create device" after completing configuration).
- For icon toggle buttons, limit labels to 2–3 words; use verbs first (e.g., "Hide / View", "Favorite / Unfavorite", "Like / Dislike").
- Use sentence case capitalization; avoid articles such as "the" or "a" in toggle button labels.

## Accessibility

### Keyboard interaction

| Key           | Action                                                   |
| ------------- | -------------------------------------------------------- |
| Enter / Space | Activates the button and triggers its `onClick` handler. |
| Tab           | Moves focus to the next focusable element.               |
| Shift + Tab   | Moves focus to the previous focusable element.           |

### ARIA

| Attribute       | Value              | When to apply                                                                                            |
| --------------- | ------------------ | -------------------------------------------------------------------------------------------------------- |
| `aria-label`    | Descriptive string | When the button has no visible text label (icon-only buttons).                                           |
| `a11yTitle`     | Descriptive string | HPE Design System prop equivalent to `aria-label`; use for icon-only buttons.                            |
| `aria-disabled` | `"true"`           | When the button is in a disabled state.                                                                  |
| `role`          | `button`           | Applied automatically by the browser for native `<button>` elements; verify when using a custom element. |

### Buttons vs. anchors

Use a `Button` when an action happens on the page — submitting a form, opening a dialog, or triggering a state change. Use an `Anchor` (or a native `<a>`) when the user navigates to a new location.

See [Anchor WCAG compliance](/components/anchor#wcag-compliance) for the related accessibility rules.

### Screen reader announcements

When a Button is activated, the screen reader announces the visible label text or the `a11yTitle` value for icon-only buttons. Ensure every button has a meaningful, action-oriented label or `a11yTitle`.
