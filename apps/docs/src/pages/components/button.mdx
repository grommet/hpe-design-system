## Variants

### Primary button

Used for the single highest-value action on a page — the one thing you most want the user to do. Only one primary button should appear per view. All other supporting actions should use a default button. Consider that many enterprise pages are primarily for viewing, not acting; in those cases a secondary button may be sufficient.

### Button with badge

The `badge` prop adds a small indicator overlay to convey extra information such as a notification count or attention signal.

- Pass `true` to display a simple dot indicating something is new or needs attention.
- Pass a number to show a count. If the value exceeds the max (default: 9), the badge displays "+" appended to the max (e.g., 9+).

## Behaviors and States

### Interactive States

Rest, hover, focus, and active states are provided by the Grommet theme and require no additional implementation. The `active` prop may also be set explicitly to mark the currently selected item in a toggle group.

### Application States

#### Disabled

Use `disabled` to prevent interaction when prerequisites have not been met. Prefer leaving buttons active with contextual feedback over disabling them — disabled controls tend to prompt "why?" questions from users. Buttons that do not apply due to context or permissions should be hidden entirely rather than shown as disabled.

#### Busy

Use `busy` to indicate a short async operation triggered by the button is in progress (a few seconds, such as server-side form validation). Display a busy state to reassure the user that their action was received.

If the operation will likely take longer than a few seconds, do not use the busy state. Instead allow the user to navigate away and notify them via another mechanism when the task completes.

#### Responsiveness

Always design for all browser widths and resolutions. Apply the following rules regardless of whether mobile or tablet use is anticipated:

- Button labels must not wrap, truncate, or scale down on smaller viewports.
- The minimum button width for any button is 42px.

## Content Guidelines

### Label wording

Labels should clearly communicate the action the button performs. Follow these rules:

- Start with a verb that describes the action (e.g., "Save changes", "Delete device").
- Use sentence case capitalization; avoid title case.
- Avoid contractions or colloquialisms. For tone guidance, see [point of view](https://design-system.hpe.design/foundation/voice-and-tone?q=tone#point-of-view).
- Ensure the label matches the route destination or the resulting action.
- Limit labels to 2–4 words where possible.

### When to use "Add," "New," and "Create"

| Label | Purpose | Example |
| --- | --- | --- |
| **Add** | An object already exists; the action builds an association between that object and another. | "Add device" — associates an existing device with a group. |
| **New** | Opens a blank template for user input before any record is created. | "New device" — starts the configuration process. |
| **Create** | Submits user input to persist the object in the system. | "Create device" — submits the form and creates the record. |

### Toggle button labels

For icon-only toggle buttons, the accessible label describes what will happen when clicked, not the current state:

- Limit to 2–3 words starting with a verb (e.g., "Hide menu", "Favorite item", "Switch theme").
- Avoid filler words such as "the" or "a".
- Use sentence case.

## Accessibility

The following guidance is the responsibility of the implementer. WCAG criteria provided directly by the Grommet component are summarized under [WCAG compliance](#wcag-compliance).

### Keyboard interaction

| Key | Action |
| --- | --- |
| Space | Activates the button. |
| Enter | Activates the button. |
| Tab | Moves focus to the next focusable element. |
| Shift + Tab | Moves focus to the previous focusable element. |

### ARIA

- **Icon-only buttons:** Supply `a11yTitle` (Grommet prop) or `aria-label` to give screen reader users a meaningful description. The label should describe the resulting action, not the icon name.
- **Toggle buttons:** Update `a11yTitle` or `aria-label` dynamically to reflect the action that will occur next (e.g., "Hide menu" when menu is visible, "Show menu" when hidden), not the current state.
- If pressing a Button results in a new URL or a new browser context, use `<Anchor>` instead. Screen readers announce buttons and links differently; the wrong element misleads users about what will happen next.
