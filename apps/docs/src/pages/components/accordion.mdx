## Use cases

### Progressively disclosing detailed content

Use Accordion when a page contains multiple sections of supporting detail
that most users don't need to read in full. Collapsing content by default
keeps the page scannable and lets users choose what to expand.

<Example height={{ min: 'small' }}>
  <AccordionProgressivelyDisclosingExample />
</Example>

### Organizing reference content into named sections

Use Accordion to divide a long reference page — such as a settings panel,
help article, or configuration guide — into clearly labeled sections.
Users can jump directly to the section they need without scrolling past
irrelevant content.

<Example height={{ min: 'small' }}>
  <AccordionOrganizingReferenceContentExample />
</Example>

### Reducing page length without sacrificing content access

Use Accordion when vertical space is constrained but all content must
remain accessible. Collapsed panels preserve access to every section
while keeping the visible page short.

## Behaviors and States

### Interactive states

- **Rest** — panel header at its default appearance; chevron points down.
- **Hover** — subtle background change to indicate the header is interactive.
- **Focus** — visible focus ring on the panel header for keyboard navigation.
- **Active** — pressed state during click or keyboard activation.

### Application states

- **Collapsed** — default state; panel content is hidden and the chevron points down.
- **Expanded** — panel content is visible and the chevron points up.

## Content Guidelines

- Keep panel labels short and descriptive — they are the user's primary navigation aid when all panels are collapsed.
- Avoid lengthy panel body content. If a panel's content requires significant scrolling, consider restructuring the page.
- Use sentence case for panel labels.
- Never use labels like "Click here" or "More info" — labels must communicate the topic of the content.

## Accessibility

Use descriptive `label` values on each `AccordionPanel` — the label is
announced by screen readers as the button name, and is the user's only
context for what the panel contains before expanding it. Buttons are used
for accordion panel headers, making them keyboard and screen reader
accessible by default.

To aid readability, allow only one panel to be open at a time unless the
user has a clear need to compare content across panels.
