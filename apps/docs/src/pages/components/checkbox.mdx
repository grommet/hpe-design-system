## Use Cases

### Accepting an agreement

The user reads a statement — such as a privacy policy, terms of service, or disclaimer — and confirms acceptance by selecting a checkbox. Place the checkbox standalone, outside a `Form` with a submit action, in account setup steps or modal confirmations. Write the label as a first-person declaration, for example, "I agree to the terms and conditions". Selecting the checkbox enables a continue or submit button.

### Revealing content conditionally

Use a checkbox to show or hide additional fields or sections based on the user's choice. This keeps the form compact by surfacing detail only when relevant. The label should make the consequence of checking the box clear, for example, "Ship to a different address".

Use a parent checkbox with `indeterminate` support to let users select all, some, or none of a set of items. The parent reflects the mixed state when only a subset of children are checked.

## Behaviors and States

### Interactive states

- **Rest** — default, unfocused appearance.
- **Hover** — a visible hover indicator appears when the pointer is over the control or label.
- **Focus** — a focus ring appears around the control when navigating via keyboard.
- **Active** — the control appears pressed while holding the mouse button or Space key.

### Application states

**Disabled** — prevents interaction when a choice is unavailable. Always communicate why a checkbox is disabled when the reason is not obvious from context.

**Indeterminate** — indicates that only a subset of child checkboxes in a group are selected. Use indeterminate only on a parent checkbox that controls a list of child checkboxes.

**Validation error** — appears via the enclosing `FormField` when the checkbox fails form validation.

## Content guidelines

- Use concise, action-oriented label text that clearly describes what selecting the checkbox means.
- Avoid double negatives in labels. Instead of "Do not disable notifications," write "Enable notifications."
- When used in a group, ensure each label is distinct and does not merely repeat the group heading.

## Accessibility

CheckBox renders a native `<input type="checkbox">` element. The `label` prop provides the accessible name. Always provide a clear, descriptive label for every CheckBox.

Never set a `label` prop on an enclosing `FormField` when the CheckBox already has a `label` — two associated `<label>` elements on one input cause accessibility violations and screen reader confusion.

When rendering a CheckBox without a `label` prop, always provide an `aria-label` or `aria-labelledby` that describes the checkbox purpose.

### Keyboard interactions

| Key          | Action                                         |
| ------------ | ---------------------------------------------- |
| Space        | Toggles the checked state of a focused checkbox. |
| Tab          | Moves focus to the next focusable element.      |
| Shift + Tab  | Moves focus to the previous focusable element. |

### ARIA

| Attribute         | Value                    | Condition                                                  |
| ----------------- | ------------------------ | ---------------------------------------------------------- |
| `aria-checked`    | `true \| false \| mixed` | Reflects checked, unchecked, or indeterminate state.       |
| `aria-label`      | descriptive string       | When no visible `label` prop is provided.                  |
| `aria-labelledby` | id of labeling element   | When a visible label element exists outside the component. |
| `aria-disabled`   | `true`                   | When the `disabled` prop is `true`.                        |
